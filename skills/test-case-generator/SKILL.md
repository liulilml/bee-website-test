# 需求驱动测试用例生成器

## 概述

从需求输入（文档/描述/设计稿图片）出发，经过 AI 分析 → 测试点拆分 → 交互确认 → 用例生成，
最终产出可直接使用的测试点文档和禅道导入文件。

---

## ⚙️ 配置

### 配置文件

路径：`<skill目录>/config.json`

```json
{
  "feishu_folder_token": "",
  "product_name": "",
  "module_name": "",
  "creator": "李茂林"
}
```

### 配置项说明

| 配置项 | 必填 | 说明 |
|--------|------|------|
| `feishu_folder_token` | ✅ | 飞书云盘文件夹 Token，测试点文档和用例文件存放位置 |
| `product_name` | 可选 | 禅道产品名，格式如 `Bee-Flutter3(#24)`。可每次输入时指定 |
| `module_name` | 可选 | 禅道模块名，格式如 `/会员订阅(#380)`。可每次输入时指定 |
| `creator` | ✅ | 创建者姓名，固定值 `李茂林` |

### 首次使用

1. 读取 `<skill目录>/config.json`
2. 如果 `feishu_folder_token` 为空，询问用户提供飞书文件夹链接
3. 配置就绪后继续

## 完整工作流程

### 🔵 阶段一：需求输入与解析

#### 支持的输入类型

| 输入类型 | 处理方式 |
|----------|---------|
| 需求文档（飞书文档/本地文件） | `feishu_fetch_doc` 或 `read` 读取内容 |
| 文字描述 | 直接从对话获取 |
| 设计稿图片 | `image` 工具识别 UI 元素、交互流程、业务规则 |
| 混合输入 | 以上任意组合 |

#### 解析目标

从输入中提取：
- **功能模块**：系统包含哪些功能区域
- **业务规则**：每个功能的具体逻辑（条件、限制、计算公式等）
- **用户角色**：涉及哪些角色，权限差异
- **状态流转**：对象的状态变化链路
- **边界条件**：数值范围、时间限制、数量上限等
- **异常场景**：网络异常、并发、数据为空等
- **UI 交互**：按钮、弹窗、表单、导航等（来自设计稿）

#### 图片解析要点

当输入包含设计稿/UI 截图时：
1. 使用 `image` 工具分析图片
2. 提取：页面元素、交互控件、文案、布局、状态展示
3. 推断：用户操作路径、输入验证规则、异常提示
4. 结合文字需求交叉验证

### 🟡 阶段二：测试点拆分

#### 测试点结构

按以下层级组织测试点：

```markdown
# [功能名称] 测试点

## 1. [模块/功能区域]

### 1.1 [子功能]

#### 正向测试
- [ ] TP-001: [测试点描述]
- [ ] TP-002: [测试点描述]

#### 反向测试
- [ ] TP-003: [测试点描述]

#### 边界测试
- [ ] TP-004: [测试点描述]

#### 兼容性/特殊场景
- [ ] TP-005: [测试点描述]

### 1.2 [子功能]
...

## 2. [模块/功能区域]
...

---

## 覆盖统计
- 正向测试点：X 个
- 反向测试点：X 个
- 边界测试点：X 个
- 特殊场景：X 个
- **总计：X 个**
```

#### 测试点设计方法论

**等价类划分：**
- 有效等价类：符合规则的典型输入
- 无效等价类：违反规则的各类异常输入

**边界值分析：**
- 最小值、最小值-1、最小值+1
- 最大值、最大值-1、最大值+1
- 空值、零值

**状态转换：**
- 每个合法状态转换路径
- 非法状态转换尝试

**场景法：**
- 主流程（Happy Path）
- 分支流程
- 异常中断与恢复

#### 测试点命名规范

```
TP-[模块缩写]-[序号]: [操作对象] + [操作动作] + [预期结果/验证点]
```

示例：
- `TP-VIP-001: 未订阅用户点击订阅按钮，正确跳转支付页`
- `TP-VIP-002: 已订阅用户查看权益页，显示正确的到期时间`
- `TP-VIP-003: 会员过期后访问专属功能，提示续费`

### 🟢 阶段三：交互确认

这是**核心阶段**，必须与用户充分沟通，不能跳过。

#### 确认流程

1. **生成测试点文档**
   - 使用 `feishu_create_doc` 创建飞书文档
   - 文件夹：`config.json` 的 `feishu_folder_token`
   - 标题：`[功能名] 测试点 - YYYY-MM-DD`
   - 内容：阶段二产出的 Markdown 测试点

2. **通知用户确认**
   - 在对话中发送文档链接
   - 提示用户可以：
     - 在飞书文档中**批注**（评论具体测试点）
     - 在飞书对话中**直接沟通**修改意见

3. **等待用户反馈**
   - 用户可能说："确认了"、"没问题"、"可以生成用例了" → 进入阶段四
   - 用户可能说："XX 测试点要改"、"少了 XX 场景"、"这个不需要" → 执行修改
   - 用户可能在飞书文档中批注 → 读取批注内容（`feishu_doc_comments`），执行修改

4. **修改测试点**
   - 根据用户反馈使用 `feishu_update_doc` 更新文档
   - 回复用户确认修改内容
   - 重复步骤 3-4 直到用户确认定稿

5. **定稿确认**
   - 用户明确表示"可以了"/"确认"/"生成用例吧" → 进入阶段四

#### ⚠️ 重要约束

- **不能自动跳过确认**：必须等用户明确确认后才进入用例生成
- **修改必须更新文档**：不能只在对话里改，飞书文档必须同步更新
- **保留修改记录**：在文档底部记录修改历史

### 🔴 阶段四：用例生成

#### 用例展开规则

基于确认的每个测试点，按以下规则展开为详细测试用例：

**一个测试点 → 一个或多个测试用例**

| 测试点类型 | 展开策略 |
|-----------|---------|
| 正向基础功能 | 1 个用例，P1 优先级 |
| 正向复杂流程 | 2-3 个用例（不同路径），P1-P2 |
| 反向/异常 | 每个异常场景 1 个用例，P2-P3 |
| 边界值 | 每个边界点 1 个用例，P2-P3 |
| 兼容性/特殊 | 按场景 1 个用例，P3-P4 |

#### 实现步骤

1. **准备测试点 JSON 文件**
   
   将确认的测试点整理为 JSON 格式，保存到 `/tmp/test_points.json`：
   ```json
   {
     "test_points": [
       {
         "id": "TP-001",
         "title": "未订阅用户点击订阅按钮",
         "type": "正向",
         "precondition": "用户未登录",
         "steps": ["1. 打开会员页", "2. 点击订阅按钮"],
         "expected": ["1. 显示登录弹窗", "2. 跳转支付页"],
         "priority": 2
       },
       ...
     ]
   }
   ```

2. **执行 CSV 生成脚本**
   
   ```bash
   python scripts/generate_csv.py \
     /tmp/test_points.json \
     /tmp/testcases_[功能名]_[日期].csv \
     "[产品名 (#ID)]" \
     "[/模块名 (#ID)]" \
     "李茂林"
   ```

3. **脚本输出**
   - CSV 文件自动生成（UTF-8 with BOM）
   - 控制台打印统计信息（总数、优先级分布）

4. **验证 CSV 格式**
   
   执行快速检查：
   ```bash
   # 检查字段数（应为 28）
   head -1 /tmp/testcases_xxx.csv | awk -F',' '{print NF}'
   
   # 检查编码（应包含 BOM）
   file /tmp/testcases_xxx.csv
   ```

#### 用例编写规范

**步骤要求：**
- 具体、可执行、无歧义
- 测试数据必须具体（不用占位符）
- 每步都有对应的预期结果

**预期结果要求：**
- 可观测、可判定（Pass/Fail）
- 描述具体的 UI 表现、数据变化、系统行为

**优先级定义：**
| 级别 | 含义 | 适用 |
|------|------|------|
| 1 | 最高 | 核心主流程，阻塞性验证 |
| 2 | 高 | 重要功能、常见异常 |
| 3 | 中 | 边界条件、次要流程 |
| 4 | 低 | 极端场景、兼容性 |

**关键词：**
- 每个用例提取 2-4 个关键词，逗号分隔
- 用于禅道搜索和分类

### 🏁 阶段五：产出与交付

#### 产出物 1：测试点文档（飞书）

阶段三已生成，此时为最终确认版。

#### 产出物 2：禅道 CSV 文件

**CSV 生成脚本**：`scripts/generate_csv.py`

**执行方式**：
```bash
python scripts/generate_csv.py <测试点 JSON> <输出 CSV> <产品名> <模块名> [创建者]
```

**CSV 格式（严格遵循禅道开源版导入规范）：**

```csv
"用例编号","所属产品","所属模块","相关软件需求","所属场景","用例标题","前置条件","步骤","预期","实际情况","关键词","优先级","用例类型","适用阶段","用例状态","B","R","S","执行人","执行时间","结果","由谁创建","创建日期","最后修改者","修改日期","用例版本","相关用例","附件"
```

**字段填写规则：**

| 字段 | 填写规则 |
|------|---------|
| 用例编号 | 从 1 开始自增（占位，禅道导入后自动分配） |
| 所属产品 | 从 config 或用户输入获取，格式：`产品名(#ID)` |
| 所属模块 | 从 config 或用户输入获取，格式：`/模块名(#ID)` |
| 相关软件需求 | 留空 |
| 所属场景 | `/(#0)` |
| 用例标题 | 简洁描述，动宾结构 |
| 前置条件 | 执行用例前必须满足的条件 |
| 步骤 | `1. xxx\n2. xxx` 格式，换行分隔 |
| 预期 | `1. xxx\n2. xxx` 格式，与步骤一一对应 |
| 实际情况 | 步骤对应的空编号：`1. \n2. ` |
| 关键词 | 逗号分隔的关键词 |
| 优先级 | 1-4 数字 |
| 用例类型 | `功能测试` |
| 适用阶段 | `功能测试阶段` |
| 用例状态 | `正常` |
| B | `0` |
| R | `0` |
| S | 步骤数量（数字） |
| 执行人 | 留空 |
| 执行时间 | 留空 |
| 结果 | 留空 |
| 由谁创建 | `李茂林`（从 config 读取） |
| 创建日期 | 当天日期 `YYYY-MM-DD` |
| 最后修改者 | 留空 |
| 修改日期 | 留空 |
| 用例版本 | `1` |
| 相关用例 | 留空 |
| 附件 | 留空 |

**CSV 生成要求：**
- 编码：UTF-8 with BOM（禅道要求）
- 所有字段用双引号包裹
- 换行符在字段内用实际换行（`\n`）
- 使用 `exec` 写入文件到 `/tmp/testcases_[功能名]_[日期].csv`

#### 产出物 3：上传到飞书

**实现步骤：**

1. **上传 CSV 到飞书云盘**
   ```
   feishu_drive_file upload
   - file_path: /tmp/testcases_[功能名]_[日期].csv
   - folder_token: [config.feishu_folder_token]
   ```

2. **在对话中发送交付物**
   
   使用 `message` 工具发送：
   - 测试点文档链接（阶段三已创建）
   - CSV 文件下载链接（刚上传的）
   - 用例统计信息：
     ```
     📊 测试用例生成完成
     
     总计：X 个用例
     - P1: X 个（核心主流程）
     - P2: X 个（重要功能）
     - P3: X 个（边界条件）
     - P4: X 个（极端场景）
     
     覆盖率：
     - 正向测试：X 个
     - 反向测试：X 个
     - 边界测试：X 个
     - 特殊场景：X 个
     
     📎 下载链接：[CSV 文件链接]
     📝 测试点文档：[飞书文档链接]
     ```

3. **可选：创建飞书任务**
   
   如果用户需要跟踪测试执行，可以创建飞书任务：
   ```
   feishu_task_task create
   - summary: [功能名] 测试用例执行
   - description: 共 X 个用例，P1: X, P2: X, P3: X, P4: X
   - due: [用户指定的截止时间]
   - members: [{id: "ou_xxx", role: "assignee"}]
   ```

## 快速命令

用户可以通过以下方式触发不同阶段：

| 用户说的 | 触发行为 |
|---------|---------|
| "帮我拆测试点" + 需求 | 阶段一 → 阶段二 → 阶段三 |
| "测试点确认了" / "可以生成用例" | 阶段四 → 阶段五 |
| "修改测试点 XX" | 阶段三（修改循环） |

| "换个产品/模块" | 更新 config 中的 product_name / module_name |

## 示例流程

### 用户输入
```
帮我拆测试点，需求是会员订阅功能：
- 产品：Bee-Flutter3(#24)
- 模块：/会员订阅(#380)
- 需求文档：[飞书链接]
- 设计稿：[图片]
```

### AI 执行
1. 读取飞书文档 + 分析设计稿图片
2. 提取功能点、业务规则、边界条件
3. 生成测试点 Markdown → 创建飞书文档
4. 发送链接给用户："测试点文档已生成，请确认：[链接]"
5. 等待用户确认（批注/对话）
6. 用户说"确认了" → 展开测试用例 → 生成 CSV
7. 上传 CSV 到飞书 → 发送下载链接

## 质量检查清单

生成用例前必须确认：
- [ ] 所有测试点都已被用户确认
- [ ] 每个测试点至少有一个对应用例
- [ ] 正向/反向/边界场景覆盖完整
- [ ] 步骤数量与预期数量一致
- [ ] 测试数据具体，无占位符
- [ ] CSV 格式符合禅道导入要求（28 字段、UTF-8 BOM）
- [ ] 已上传到飞书并发送链接
